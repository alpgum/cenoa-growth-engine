# Marketing Autopilot v1.0 — Launch Checklist (KPIs → Anomalies → Health → Actions → Weekly Report)

Scope: the Monday automation that refreshes the KPI snapshot, flags anomalies, runs campaign health + data quality guardrails, generates weekly actions, and sends the weekly pulse report.

**System map (artifacts + owners):**
- KPI snapshot (Amplitude → `projects/cenoa-cortex/data.json`) — Data Lead
- Anomalies (`projects/cenoa-performance-marketing/data/anomalies.json`) — Data Lead
- Campaign health (`projects/cenoa-performance-marketing/data/campaign-health.json`) — Acquisition + Data
- Weekly actions (`projects/cenoa-performance-marketing/data/weekly-actions.json`) — Data Lead drafts; Acquisition/Product finalize
- Weekly report (stdout → Telegram + `data/weekly-report-latest.md`) — Data Lead
- Guardrail: Data quality monitor (`data/data-quality.json` + `analysis/data-quality-monitoring.md`) — Data Lead

### 2.3 Create cron jobs (OpenClaw Gateway scheduler)

All schedules below assume **Europe/Istanbul** timezone.

> Recommended: add jobs first, run them once manually (via `openclaw cron run`), then enable.

**(A) Weekly KPI refresh + Cortex deploy**
- Script: `scripts/weekly_kpi_cron.sh`
- Purpose: writes `projects/cenoa-cortex/data.json`, commits/pushes, deploys to Vercel.

**(B) Anomaly alert**
- Script: `scripts/anomaly_alert_cron.sh`
- Purpose: writes `data/anomalies.json`; prints Telegram HTML only if anomalies exist.

**(C) Campaign health sweep**
- Script: `scripts/dead_campaign_sweep_cron.sh`
- Purpose: refreshes `data/campaign-health.json`; prints Telegram HTML only if DEAD/BLEEDING/FRAUD exists.

**(D) Weekly report (includes weekly_actions best-effort)**
- Script: `scripts/weekly_report_cron.sh`
- Purpose: generates report to stdout + saves `data/weekly-report-latest.md`.

**(E) Data quality monitor (guardrail)**
- Script: `scripts/data_quality_monitor.py`
- Purpose: writes `data/data-quality.json` + `analysis/data-quality-monitoring.md`; non-zero exit on WARN/CRIT.

Example cron add commands (adjust channels/targets as needed):
```bash
# KPI refresh
openclaw cron add --schedule "0 9 * * 1" --tz "Europe/Istanbul" \
  --command "cd ~/.openclaw/workspace/projects/cenoa-performance-marketing && bash scripts/weekly_kpi_cron.sh"

# Anomaly detection (run AFTER KPI refresh)
openclaw cron add --schedule "20 9 * * 1" --tz "Europe/Istanbul" \
  --command "cd ~/.openclaw/workspace/projects/cenoa-performance-marketing && bash scripts/anomaly_alert_cron.sh" \
  --channel telegram

# Campaign health (after KPI refresh)
openclaw cron add --schedule "25 9 * * 1" --tz "Europe/Istanbul" \
  --command "cd ~/.openclaw/workspace/projects/cenoa-performance-marketing && bash scripts/dead_campaign_sweep_cron.sh" \
  --channel telegram

# Data quality monitor (guardrail; see failure handling section)
openclaw cron add --schedule "30 9 * * 1" --tz "Europe/Istanbul" \
  --command "cd ~/.openclaw/workspace && source ~/.openclaw/venv/bin/activate && \
GOOGLE_APPLICATION_CREDENTIALS=~/.openclaw/credentials/cenoa-marketingdatawarehouse-82b8600e66d6.json \
python3 projects/cenoa-performance-marketing/scripts/data_quality_monitor.py"

# Weekly report (includes weekly_actions.py best-effort)
openclaw cron add --schedule "0 10 * * 1" --tz "Europe/Istanbul" \
  --command "cd ~/.openclaw/workspace/projects/cenoa-performance-marketing && bash scripts/weekly_report_cron.sh" \
  --channel telegram
```

**Guardrail recommendation:** keep the report cron enabled, but treat the report as **non-decision-grade** if data quality is **CRIT**.
If you want hard gating, wrap `weekly_report_cron.sh` with a check on `data/data-quality.json` (CRIT → exit 0 without posting) and schedule that wrapper instead.

## 5) Failure handling (especially Data Quality monitor WARN/CRIT)

### 5.1 Severity meaning (convention)
- **PASS**: safe to use outputs for decisions.
- **WARN**: outputs likely usable but **require skepticism**; investigate within 24–48h.
- **CRIT**: outputs are **not decision-grade**; treat as incident; fix before acting.

### 5.2 What to do if Data Quality is **CRIT**

1) Open the human-readable report:
   - `projects/cenoa-performance-marketing/analysis/data-quality-monitoring.md`
2) Identify which check failed (examples below) and take the matching action.
3) Re-run the failing upstream pipeline(s).
4) Re-run `data_quality_monitor.py` until overall severity improves to at least WARN.
5) **Guardrail:** do not pause/scale budgets based solely on the automated weekly report while CRIT.

Common CRIT causes + responses:

- **Missing required files** (example: `data.json` missing)
  - Cause: KPI cron didn’t run / failed / ran in wrong working dir.
  - Fix:
    - Re-run: `bash scripts/weekly_kpi_cron.sh`
    - Confirm: `~/.openclaw/workspace/projects/cenoa-cortex/data.json` exists.

- **Amplitude “(none)” country/platform share too high**
  - Symptom: signups have huge `(none)` share in `amplitude-country-*` / `amplitude-platform-*`.
  - Likely causes: missing user properties, server-side events without device context, web→app handoff losing fields.
  - Fix path:
    - Re-export country/platform breakdowns:
      ```bash
      source ~/.openclaw/venv/bin/activate
      python3 projects/cenoa-performance-marketing/scripts/amplitude_country_breakdown.py
      python3 projects/cenoa-performance-marketing/scripts/amplitude_platform_breakdown.py
      ```
    - Escalate to Product/Data Eng: instrumentation mapping + SDK init + event property contracts.

### 5.3 What to do if Data Quality is **WARN**

- Proceed with the weekly report, but add an explicit note in Monday review: “Data quality WARN: <summary>”.
- Prioritize fixes if WARN persists 2+ consecutive weeks.

Common WARN causes + responses:

- **Organic installs share high while paid spend proxy is high**
  - Likely: paid→organic attribution leakage.
  - Actions:
    - Cross-check AppsFlyer vs platform dashboards.
    - Audit OneLink / deep link config and UTM persistence.

- **BigQuery daily table coverage low**
  - Actions:
    - Backfill partitions for `marketing_appsflyer.daily_installs_campaign_tr`.
    - Verify scheduled load/ELT job is running and permissions are intact.

### 5.4 Component-level failures (quick triage)

- `weekly_kpi_cron.sh` fails
  - Check: Amplitude credentials, network, git auth, Vercel auth/link.
  - Immediate recovery: run `kpi_auto_update.py --dry-run` to isolate Amplitude vs deploy failures.

- `weekly_report_cron.sh` fails because `data.json` missing
  - Root cause is always upstream (KPI refresh). Fix KPI first.

- `anomaly_alert_cron.sh` produces no message
  - Normal when anomaly count = 0.
  - Confirm file exists: `data/anomalies.json`.

- `dead_campaign_sweep_cron.sh` produces no message
  - Normal when no DEAD/BLEEDING/FRAUD.
  - Confirm JSON refreshed: `data/campaign-health.json`.

## 6) Rollback plan (disable crons + revert changes)

### 6.1 Disable automation immediately

List jobs:
```bash
openclaw cron list
```

Disable the relevant jobs (repeat per id):
```bash
openclaw cron disable <job_id>
```

If you need to remove completely:
```bash
openclaw cron rm <job_id>
```

### 6.2 Revert a bad Cortex data publish (data.json)

In `projects/cenoa-cortex`:
```bash
cd ~/.openclaw/workspace/projects/cenoa-cortex

git log -n 5 --oneline
# Option A: revert the bad commit safely
git revert <bad_commit_sha>

git push
npx vercel --prod --yes --force
```

> Avoid `git reset --hard` unless you’re 100% sure nobody else depends on the history.

### 6.3 Revert performance-marketing scripts/config

```bash
cd ~/.openclaw/workspace/projects/cenoa-performance-marketing

git log -n 10 --oneline
git revert <bad_commit_sha>
```
